# Feature Specification: API Parity Verification

**Feature Branch**: `008-api-parity`
**Status**: Draft
**Created**: 2026-04-09

## Compatibility Context *(mandatory)*

- **Salesforce Surface**: REST, Bulk API v2, Metadata API, Tooling API, OAuth2
- **Compatibility Target**: Real Salesforce org "dev 20" via `sf` CLI
- **In-Scope Operations**: Capture real Salesforce responses, compare with sf_localstack responses, produce structured gap report
- **Out-of-Scope Operations**: Fixing identified gaps (deferred to F6 API Fidelity), SOAP APIs beyond Metadata describe/list/read, UI or dashboard parity
- **Test Isolation Plan**: Parity tests create temporary records with the `PARITY_TEST_` prefix in the dev 20 org. All temporary records are deleted in a cleanup step at the end of each test run, regardless of test outcome.
- **Parity Verification Plan**: This feature IS the parity verification mechanism. It establishes the baseline diff between real Salesforce and sf_localstack, produces a categorized gap report, and provides a re-runnable framework to measure improvement over time.

## Feature Iterations *(mandatory)*

### Feature 0 - Parity Test Framework (Priority: P1)

Create the `parity-tests/` directory with shell scripts, Java comparison utilities, and output report templates. The framework must capture, normalize, and diff API responses from two targets — dev 20 and sf_localstack — and emit a structured per-test-case diff plus a summary report.

**Why this priority**: All subsequent iterations depend on the framework. Nothing can be compared without it.

**Comparison scope**:
- HTTP status code
- Response headers: `Content-Type`, `Sforce-Limit-Info`
- Response body shape: JSON key names, nesting structure, field data types (string, boolean, number, array, object, null), required vs optional presence
- CSV structure for Bulk result responses: column headers, row format

**Normalization rules** (applied before diff):
- Dynamic values ignored: IDs (18-char alphanumeric), timestamps (ISO 8601 dates), organization-specific strings matching a known-dynamic list
- Key ordering normalized: JSON objects sorted alphabetically before comparison
- Array element ordering normalized where order is not semantically significant
- Numeric precision differences flagged as Minor, not Critical

**Output**:
- Per-test-case diff: `parity-tests/reports/<test-name>.diff.json` — machine-readable JSON with `{ matched: [...], missing_from_emulator: [...], extra_in_emulator: [...], type_mismatches: [...] }`
- Summary report: `parity-tests/reports/gap-report.md` — human-readable markdown (generated by F5)

**Acceptance Scenarios**:

1. **Given** the framework exists and both targets are reachable, **When** a parity test runs, **Then** a `.diff.json` file is written to `parity-tests/reports/` with a structured comparison result.
2. **Given** two responses that are structurally identical but differ only in IDs and timestamps, **When** the comparison runs, **Then** the diff reports zero structural mismatches.
3. **Given** dev 20 is unreachable, **When** a parity test runs, **Then** the framework exits with a non-zero status code and prints a clear prerequisite error message (`ERROR: cannot reach Salesforce org 'dev 20' — is sf CLI authenticated?`).
4. **Given** sf CLI is not installed or `sf` is not on PATH, **When** a parity test runs, **Then** the framework detects this in a prerequisite check and exits before attempting any API calls.

### Feature 1 - REST API Parity Tests (Priority: P1)

Test the full REST API surface using `sf data` and `sf sobject` commands. Run identical operations against dev 20 and sf_localstack, capture responses, and produce per-operation diffs.

**Why this priority**: REST CRUD and SOQL are the most heavily used surfaces by sf CLI tools. Any gap here breaks the most common developer workflows.

**Test cases**:

| Test ID | Operation | sf CLI Command |
|---|---|---|
| REST-01 | Create Account record | `sf data create record --sobject Account --values "Name='PARITY_TEST_Account'"` |
| REST-02 | Get record by ID | `sf data get record --sobject Account --record-id <id-from-REST-01>` |
| REST-03 | Update record | `sf data update record --sobject Account --record-id <id-from-REST-01> --values "Name='PARITY_TEST_Account_Updated'"` |
| REST-04 | SOQL query | `sf data query --query "SELECT Id, Name FROM Account WHERE Name LIKE 'PARITY_TEST_%'"` |
| REST-05 | Delete record | `sf data delete record --sobject Account --record-id <id-from-REST-01>` |
| REST-06 | Describe sObject | `sf sobject describe --sobject Account` |
| REST-07 | List sObjects | `sf sobject list` |
| REST-08 | Upsert by external ID | Direct REST call: `PUT /services/data/v60.0/sobjects/Account/ExternalId__c/PARITY_TEST_EXT_001` |

**Comparison focus**: response body shape (key names, nesting, field types), HTTP status codes, error format when record not found (404 shape), pagination envelope for SOQL query results.

**Acceptance Scenarios**:

1. **Given** both targets are running, **When** REST-01 through REST-08 run, **Then** a diff report is produced for each test case showing matches and mismatches.
2. **Given** REST-06 (describe) runs, **When** the diff is computed, **Then** the key structure of the describe response (top-level keys: `name`, `fields`, `recordTypeInfos`, `childRelationships`) is compared field by field with type mismatches flagged.
3. **Given** records created with `PARITY_TEST_` prefix, **When** the test run completes (success or failure), **Then** all temporary records are deleted from dev 20 in the cleanup step.
4. **Given** dev 20 returns a field not present in the emulator's Account model, **When** the diff runs, **Then** that field is reported as `missing_from_emulator` in the diff, and the test continues without failing the framework.

### Feature 2 - Bulk API Parity Tests (Priority: P1)

Test the full Bulk API v2 ingest lifecycle against both targets. Compare job state transitions, result CSV format, and error handling.

**Why this priority**: Bulk API is the primary data-load mechanism. Format divergence in CSV results or job state names breaks data loading tools.

**Test lifecycle** (run end-to-end against each target):

| Step | Operation | HTTP Method + Path |
|---|---|---|
| BULK-01 | Create ingest job | `POST /services/data/v60.0/jobs/ingest` |
| BULK-02 | Upload CSV batch | `PUT /services/data/v60.0/jobs/ingest/<jobId>/batches` |
| BULK-03 | Close job (signal upload complete) | `PATCH /services/data/v60.0/jobs/ingest/<jobId>` body: `{"state":"UploadComplete"}` |
| BULK-04 | Poll job status until terminal | `GET /services/data/v60.0/jobs/ingest/<jobId>` |
| BULK-05 | Fetch successful results | `GET /services/data/v60.0/jobs/ingest/<jobId>/successfulResults` |
| BULK-06 | Fetch failed results | `GET /services/data/v60.0/jobs/ingest/<jobId>/failedResults` |
| BULK-07 | Fetch unprocessed results | `GET /services/data/v60.0/jobs/ingest/<jobId>/unprocessedrecords` |
| BULK-08 | Delete job | `DELETE /services/data/v60.0/jobs/ingest/<jobId>` |

**CSV test data**: Upload 3 rows including one deliberately invalid row (missing required field) to verify both success and failure result formats.

**Comparison focus**:
- Job creation response JSON shape (keys: `id`, `operation`, `object`, `state`, `createdDate`, `contentType`, `apiVersion`, etc.)
- State machine values: exact string names for each state (`Open`, `UploadComplete`, `InProgress`, `JobComplete`, `Failed`, `Aborted`)
- Result CSV column headers: `sf__Id`, `sf__Created`, `sf__Error` — exact casing and spelling
- HTTP status codes for each step

**Acceptance Scenarios**:

1. **Given** both targets are running, **When** the Bulk lifecycle BULK-01 through BULK-08 runs, **Then** diff reports for job create, status, and each result type are produced.
2. **Given** the Bulk lifecycle completes on both targets, **When** the result CSVs are compared, **Then** column header names and row format are compared character-by-character (after normalizing IDs).
3. **Given** state machine polling completes, **When** the terminal state names are compared, **Then** the exact state string from dev 20 matches the emulator's state string with no casing differences.

### Feature 3 - Metadata API Parity Tests (Priority: P2)

Test Metadata API operations for describe, list, and read. Use `sf project retrieve start` and `sf project deploy start` for retrieve/deploy flow parity.

**Why this priority**: Metadata API is used by CI pipelines. Shape mismatches in describe or list responses break metadata-driven tooling.

**Test cases**:

| Test ID | Operation | Method |
|---|---|---|
| META-01 | describeMetadata | `sf` CLI or direct SOAP call |
| META-02 | listMetadata for ApexClass | Direct SOAP `listMetadata` call |
| META-03 | listMetadata for CustomObject | Direct SOAP `listMetadata` call |
| META-04 | listMetadata for Layout | Direct SOAP `listMetadata` call |
| META-05 | readMetadata for a known type | Direct SOAP `readMetadata` call |
| META-06 | Retrieve via sf CLI | `sf project retrieve start --metadata ApexClass` |
| META-07 | Deploy via sf CLI | `sf project deploy start --metadata ApexClass` |

**Comparison focus**:
- Top-level keys in `describeMetadata` response (metadata type names, `xmlName`, `directoryName`, `inFolder`, `metaFile`, `suffix`, `childXmlNames`)
- `listMetadata` result envelope: key names and array vs object for single vs multiple results
- XML response shape for `readMetadata`
- Retrieve/deploy job status field names and state values

**Acceptance Scenarios**:

1. **Given** both targets are running, **When** META-01 through META-05 run, **Then** diffs are produced comparing SOAP XML response shapes.
2. **Given** `describeMetadata` runs on both targets, **When** comparing the metadata type catalog, **Then** all types present in dev 20 but absent from the emulator are reported as gaps.
3. **Given** META-06 retrieve runs, **When** the retrieved ZIP is inspected, **Then** the directory structure and file naming conventions match real Salesforce conventions.

### Feature 4 - Error Response Parity (Priority: P2)

Deliberately trigger error conditions on both targets and compare HTTP status codes and error response JSON shapes.

**Why this priority**: sf CLI tools and Salesforce integrations parse error responses. Shape mismatches cause silent failures or wrong error messages surfaced to users.

**Error scenarios**:

| Error ID | Trigger | Expected HTTP Status |
|---|---|---|
| ERR-01 | GET non-existent sObject type: `/services/data/v60.0/sobjects/InvalidType__c/describe` | 404 |
| ERR-02 | POST Account without required Name field | 400 |
| ERR-03 | GET SOQL with syntax error: `/services/data/v60.0/query?q=INVALID+SOQL` | 400 |
| ERR-04 | GET record with malformed ID: `/services/data/v60.0/sobjects/Account/NOT_A_VALID_ID` | 404 |
| ERR-05 | Upsert conflict on duplicate external ID | 300 |
| ERR-06 | GET non-existent Bulk job ID | 400 |
| ERR-07 | PATCH Bulk job with invalid state transition | 400 |
| ERR-08 | POST with malformed JSON body | 400 |
| ERR-09 | GET with missing required query parameter | 400 |
| ERR-10 | OAuth token request with invalid client credentials | 400 |

**Comparison focus**:
- HTTP status code (exact match required)
- Error response JSON shape: top-level array vs object, key names (`message`, `errorCode`, `fields`)
- `errorCode` string values (exact string, e.g., `INVALID_TYPE`, `REQUIRED_FIELD_MISSING`, `MALFORMED_ID`)

**Acceptance Scenarios**:

1. **Given** ERR-01 through ERR-10 are triggered on both targets, **When** diffs are computed, **Then** status codes and error JSON shapes are compared for all 10 scenarios.
2. **Given** an error response comparison, **When** the `errorCode` field value differs between targets, **Then** it is flagged as a Critical gap in the diff.
3. **Given** at least 10 error scenarios are compared, **Then** SC-003 is satisfied.

### Feature 5 - Gap Report and Prioritization (Priority: P2)

Aggregate all per-test-case diffs into a single gap report at `parity-tests/reports/gap-report.md`. Categorize each gap by severity and include concrete fix suggestions.

**Why this priority**: The raw diffs from Features 1-4 are not actionable until aggregated and prioritized. This report directly feeds F5 (SOQL Engine) and F6 (API Fidelity) work.

**Gap categories**:

| Category | Definition |
|---|---|
| Critical | Breaks sf CLI command execution — wrong status code, missing required field in response, wrong state string |
| Major | Wrong response shape — missing top-level key, wrong data type, array vs object mismatch |
| Minor | Missing optional field — present in real SF but not fatal to CLI operation |
| Cosmetic | Formatting difference — whitespace, key ordering, timestamp precision |

**Report structure**:
```
parity-tests/reports/gap-report.md
├── Summary table: total gaps by category
├── Critical gaps (with fix suggestions)
├── Major gaps (with fix suggestions)
├── Minor gaps
└── Cosmetic gaps
```

**Fix suggestions**: Each Critical and Major gap must include one of:
- The exact Java file and method to modify
- The response field to add or correct
- A reference to an existing similar implementation in the codebase

**Acceptance Scenarios**:

1. **Given** all parity tests have run and produced diff files, **When** the gap report generator runs, **Then** `gap-report.md` is written with all gaps categorized by severity.
2. **Given** the gap report is complete, **When** an engineer reads it, **Then** each Critical and Major gap has a concrete fix suggestion pointing to a specific file or field.
3. **Given** the gap report is complete, **Then** it feeds directly into the F5 (SOQL Engine) and F6 (API Fidelity) spec work as a prioritized backlog.

### Edge Cases

- **What if dev 20 org is unavailable?** The framework detects this in a prerequisite check before any test runs. It prints `ERROR: cannot reach Salesforce org 'dev 20' — ensure sf CLI is installed and authenticated (sf org display -o dev20)` and exits with code 1. No partial results are written.
- **What if sf CLI is not installed?** A prerequisite check (`which sf || command -v sf`) fails early. The framework prints `ERROR: sf CLI not found on PATH — install from https://developer.salesforce.com/tools/salesforcecli` and exits with code 1.
- **What if real Salesforce returns fields not in the emulator model?** These are captured as `missing_from_emulator` entries in the diff. The test framework does not fail — it reports and continues. Missing fields are categorized as Minor gaps unless they are required by sf CLI operations (in which case they are Major or Critical).
- **What if response field ordering differs?** Both JSON objects are key-sorted before comparison. Order differences produce no diff entries.
- **What if a parity test creates records in dev 20 and then crashes before cleanup?** The framework maintains a cleanup manifest (`parity-tests/cleanup-manifest.json`) updated after each record creation. A separate `parity-tests/cleanup.sh` script reads this manifest and deletes all records listed. Cleanup can be run independently after a crash.
- **What if the emulator is not running?** The prerequisite check issues a health check request to `http://localhost:8080/services/data/v60.0/` before tests start. Failure prints `ERROR: sf_localstack not reachable at localhost:8080 — start the emulator first` and exits.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Parity tests MUST run against real Salesforce org "dev 20" via `sf` CLI using authenticated named org credentials.
- **FR-002**: Parity tests MUST run the identical operations against sf_localstack running at `localhost:8080`.
- **FR-003**: Comparison MUST normalize dynamic values (IDs matching `[a-zA-Z0-9]{15,18}`, ISO 8601 timestamps) before diffing. Structural shape differences must still be detected.
- **FR-004**: Temporary records in dev 20 MUST use the `PARITY_TEST_` prefix and MUST be deleted in a cleanup step that runs regardless of test pass/fail outcome.
- **FR-005**: Gap report MUST categorize each gap as Critical, Major, Minor, or Cosmetic using the severity definitions in Feature 5.
- **FR-006**: The framework MUST be re-runnable after fixes to measure improvement. Re-running must overwrite previous diff reports, not append to them.
- **FR-007**: The framework MUST check prerequisites (sf CLI available, dev 20 reachable, emulator reachable) before running any test, and fail fast with actionable error messages.
- **FR-008**: All parity test scripts and comparison tools MUST live under `parity-tests/` and MUST NOT modify any application source code.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Parity tests cover all supported API surfaces: REST CRUD (7 operations), SOQL (1 operation), Bulk API v2 full lifecycle (8 steps), Metadata describe/list/read (5 operations), and OAuth error (1 scenario).
- **SC-002**: `parity-tests/reports/gap-report.md` is produced after a full run, with all gaps categorized by severity and Critical/Major gaps carrying concrete fix suggestions.
- **SC-003**: At least 10 error response scenarios (ERR-01 through ERR-10) are compared and appear in the gap report.
- **SC-004**: After the test run completes (pass or fail), all `PARITY_TEST_*` records are deleted from the dev 20 org. Verified by running `sf data query --query "SELECT Id FROM Account WHERE Name LIKE 'PARITY_TEST_%'" -o dev20` and confirming zero results.
- **SC-005**: The framework can be re-run after fixes by executing a single entry-point script (`parity-tests/run-all.sh`). Re-running overwrites existing diff reports and regenerates the gap report.
